# ScottClip

Linear-backed agent orchestration for Claude Code. An Orchestrator agent triages Linear issues and infers roles — a CEO handles strategic calls, workers execute. Roles are lightweight `.md` files; labels are optional hints, not required assignments.

## How It Works

```
Linear Webhooks / /heartbeat → Orchestrator → Role Inference → Worker Dispatch → Report
```

1. **Issues** arrive via Linear webhooks (real-time) or `/heartbeat` (polling)
2. **Orchestrator** reads the inbox, infers which role owns each issue (labels help but aren't required), and pre-researches context before spawning workers
3. **Workers** execute with role context injected and skills preloaded — explore, plan, code, commit
4. **Reports** are structured Linear comments with progress, commits, blockers, and verification inference
5. **Watch mode** runs via `/scottclip-watch`; webhook sessions use the Claude Agent SDK for structured event handling

The human is the **Board** – the ultimate escalation target when the agent is blocked.

## Install

In Claude Code, run `/plugin` → **Add Marketplace** → enter `scottzilla/claude-hub`, then install the plugin.

Or for local development:

```bash
git clone https://github.com/scottzilla/scottclip.git
claude --plugin-dir /path/to/scottclip
```

### Prerequisites

- [Claude Code](https://claude.ai/code) installed
- Linear workspace with at least one team
- **`linear-agent` MCP server** (bundled at `mcp/linear-agent/`) — required. Enables delegate-based locking, agent sessions, and webhook events. Configured automatically by `/scottclip-init`. Agents reference it as `mcp__linear-agent__*`.

## Quick Start

```bash
# 1. Initialize ScottClip in your repo (creates config + roles + Linear labels)
/scottclip-init

# 2. Run a single heartbeat cycle
/heartbeat

# 3. Start watch mode (webhook receiver + polling)
/scottclip-watch

# Or just schedule polling heartbeats
/schedule 30m /heartbeat

# 4. Check status
/scottclip-status

# 5. Refresh skills after upstream updates
/sc-refresh-skills

# 6. Migrate old persona directories to roles
/sc-migrate
```

## Agent Flow

### Orchestrator

1. Read issues from Linear inbox
2. Infer role from label (or derive from issue content if unlabeled)
3. Pre-research relevant context (codebase, prior comments, related issues)
4. Spawn worker agent with role context + preloaded skills
5. Post-dispatch review — update issue state, escalate if blocked

### Worker

1. Explore repo context and issue requirements
2. Plan approach (check with Orchestrator if scope is unclear)
3. Implement — code, tests, commits
4. Report — structured comment with commits, verification results, blockers

Use `--dry-run` to see what would be picked without doing work. Use `--role backend` to force a specific role.

## Roles

Each role is a single `.md` file that defines identity, tools, and decision framework for that agent context. Roles live in `.scottclip/roles/`.

### Default Roles

| Role | Purpose | Label |
|------|---------|-------|
| Orchestrator | Triage, role inference, worker dispatch | *(default — no label needed)* |
| CEO | Strategy, prioritization, architecture | `ceo` |
| Backend | API, database, server-side implementation | `backend` |
| Frontend | UI, components, styling | `frontend` |

The Orchestrator can also compose ad-hoc roles on the fly for issues that don't fit a predefined role.

## Per-Repo Structure

After `/scottclip-init`, your repo gets:

```
.scottclip/
├── config.yaml
├── roles/
│   ├── backend.md
│   ├── frontend.md
│   └── ceo.md

    └── testing-patterns/
.claude/
└── agents/
    ├── orchestrator.md        # project-level agent
    └── worker.md              # project-level agent
```

## Commands

| Command | Description |
|---------|-------------|
| `/heartbeat` | Run one heartbeat cycle |
| `/heartbeat --dry-run` | Show what would be picked up |
| `/heartbeat --role backend` | Force a specific role |
| `/scottclip-init` | Initialize ScottClip in a repo |
| `/scottclip-status` | Current state, queue, blocked issues |
| `/scottclip-status --history` | Recent heartbeat history |
| `/scottclip-watch` | Start watch mode (webhook + polling) |
| `/scottclip-watch --interval 5m` | Custom polling interval |
| `/scottclip-watch --webhook-only` | Webhook receiver only |
| `/scottclip-watch --poll-only` | Polling heartbeat only |
| `/scottclip-watch --stop` | Stop all watch processes |
| `/sc-refresh-skills` | Pull updated skills from upstream plugin |
| `/sc-migrate` | Migrate old persona directories to role files |

## Label System

Labels are optional hints for role routing — the Orchestrator can infer role from issue content when no label is present. Workflow status is tracked via Linear's built-in states.

| Label | Purpose |
|-------|---------|
| `backend`, `frontend`, `ceo`, etc. | Hints which role should handle the issue |

**Workflow states** (Todo → In Progress → Blocked → In Review → Done) are the single source of truth for issue lifecycle. Labels are managed via read-modify-write (get labels array → modify → save full set).

## Memory

Agents use native Claude Code agent memory (`memory: project` in agent frontmatter) for persistent context across sessions. No external files or PARA structure required.

## Watch Mode & Scheduling

ScottClip supports two modes for autonomous operation:

### Watch mode (recommended)

`/scottclip-watch` starts dual-mode operation: a webhook receiver for real-time Linear agent events (using the Claude Agent SDK), plus a polling heartbeat as a reliability fallback.

```bash
/scottclip-watch                  # Both modes, 15m polling default
/scottclip-watch --interval 5m   # Faster polling
/scottclip-watch --webhook-only   # Real-time only (requires tunnel setup)
/scottclip-watch --poll-only      # Polling only (no webhook receiver)
/scottclip-watch --stop           # Stop everything
```

The webhook receiver listens for Linear agent session events (delegation, user messages) and spawns Claude agent sessions via the Agent SDK in real-time. The polling heartbeat catches anything missed and processes the broader issue queue.

**Webhook setup:** The receiver runs on port 3847. Expose it via `cloudflared tunnel --url http://localhost:3847`, then register the tunnel URL as a webhook in Linear (Settings → API → Webhooks → `AgentSessionEvent`).

### Schedule-only mode

For simpler setups without webhooks:

| Workload | Cadence | Command |
|----------|---------|---------|
| Active sprint | Every 15-30 min | `/schedule 15m /heartbeat` |
| Steady state | Every 1-2 hours | `/schedule 1h /heartbeat` |
| Background | Every 4-6 hours | `/schedule 4h /heartbeat` |
| Manual only | No schedule | `/heartbeat` when needed |

## Background

ScottClip is a substantial extension of [WoterClip](https://github.com/wotai-dev/woterclip) which is inspired by [Paperclip](https://github.com/paperclipai/paperclip), an agent orchestration platform that uses a central API for task management, agent checkout, and chain-of-command routing. ScottClip takes the same core ideas – structured heartbeats, hierarchical escalation, structured comments for audit trails – and rebuilds them as a Claude Code plugin backed by Linear. The system has evolved from 3-file persona directories to lightweight single-file roles, with the Orchestrator gaining the ability to infer roles dynamically and the Agent SDK handling structured webhook sessions.
